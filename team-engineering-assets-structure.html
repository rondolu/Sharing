team-engineering-assets

本頁為供團隊內部討論的架構提案（Refined v2），核心定位是讓開發者 clone 至本機 VS Code 後，能直接使用 Copilot 資產與開發 reference。一般文件、申請流程及表單範本一律由Microsoft Teams 公槽維護，不納入此 repo。

這份提案的核心決策

  • 此 repo 定位為開發者本機工作流，不作為一般文件倉庫。
  • .github 是可載入規則層，guides 是人類理解層，templates 是專案起手層。
  • 一般文件、申請流程及表單範本改由 Microsoft Teams 公槽維護。
  • VS Code 原生載入的 Copilot 資產統一放於根目錄 .github。

設計原則

先定義分類邊界，再思考各資產落點，最後决定資料夾命名。

工具定位優先

每份資產都應能回答：「開發者 clone 至本機後，實作時是否會直接用到」。

三層分工

.github 是可載入規則層，guides 是理解層，templates 是起手層，三層分工明確，不應混用。

避免黑洞目錄

不建立 common、shared、misc 等模糊目錄；申請流程、表單及一般團隊資訊一律改放 Microsoft Teams 公槽。

根目錄總覽

根目錄僅保留高辨識度的資產類型，讓初次開啟 repo 的開發者能快速建立心智模型。

各層用途邊界

邊界定義旨在防止未來分類漂移。

.github Load

  • 只放需要被 VS Code / Copilot 原生載入的資產：agents、skills、prompts、instructions。
  • 共通 instructions 僅放跨專案穩定不變的規則；不放每個服務都需要客製的設定。
  • 不要加包 ai-tools 或 copilot 等目錄。

guides Understand

  • 只放開發者在實作時會直接查閱的方法論與操作情境 reference。
  • .github 資產組合說明僅在多個資產需搭配理解時才在此新增；單一資產應自解釋。
  • 禁止放入：申請流程、表單範本、一般團隊資訊。

templates Copy

  • 放給開發者複製後依專案客製的起手資產。
  • API 服務需求 checklist 屬於此層；每個新服務規格不同，應複製後調整，不固化為共通 instructions。
  • 每個sub-folder都該有README.md。

最小治理要求

每個資料夾須具備自解釋能力：README 定義該層的用途邊界與使用方式，命名則反映實際用途。

README 分層架構

  • 根目錄 README：Repo 定位與分工說明、三層邊界導覽（連結各層 README）、資產放置決策流程、Copilot 資產放置規則。
  • 各層 README（guides / templates / .github）：說明該層放什麼、不放什麼。
  • 各 Blueprint README：適用情境、使用步驟、限制。

命名建議

  • 目錄一律使用 kebab-case。
  • 名稱以用途為主，不用模糊詞。
  • Blueprint 加 -base；Checklist 加 -checklist。
  • Copilot 資產名稱要對齊實際用途。
